Swarm — visual reference

Five views of the framework. Read alongside swarm-framework.md.

1. The task at the centre

The task file is the framework's source of truth. A source doc grounds it, the lead persona conditions it, skills are loaded as references, validation gates verify the work, and the output is what the task produced.

2. The thirteen task types, by output

Implementation tasks produce code. Investigation tasks produce a document. The single Communication task produces docs. Each task type has exactly one lead persona — the mapping is rigid.

3. Distillation flows downhill

Allowed flows go from broad upstream artifacts to narrow downstream ones. Reverse flow, skipping the spec, and ungrounded tasks are the three ways agents most commonly drift — the framework forbids them by name.

4. The task file's seventeen sections

Sections one, three, four, and five are pre-filled deterministically from the task type. Most others are filled by the agent during the session. Three sections (constraints, validation gates, self-review) carry deterministic skeletons that the agent then completes.

5. Worked example — research question to shipped code

The longest realistic chain. A research question becomes a research file via the Researcher; the research file becomes a spec via the Specifier; the spec becomes code via the Builder. Each arrow is one task; the persona shown on each box is who owns producing that artifact.
