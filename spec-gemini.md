Swarm Agent Documentation Framework: Architectural Blueprint and Distillation Pipelines00-overview.md: Synthesis and Ecosystem FlowThe Swarm orchestration system operates on a deterministic paradigm built for the bleeding edge of agentic software engineering. In 2026, the industry has recognized that unconstrained AI agents are prone to hallucination, context bloat, and architectural drift. Top engineering teams now rely on highly structured, multi-agent orchestrations that emphasize governance, isolated environments, and deterministic workflows over open-ended chat.This framework operationalizes the "Spec-Driven Development" (SDD) methodology. A high-level source document is progressively distilled into actionable, low-verbosity task files. These task files are subsequently injected into independent Git worktrees, automatically pre-loaded with specific Claude Code-style subagent YAML configurations, restricting their tools, models, and scopes to exact boundaries.The entire pipeline strictly enforces "Show, Don't Tell" principles, mandating empirical command-line proof before any agent execution loop terminates. Furthermore, by physically isolating concerns through a mandatory layered workflow, the framework guarantees "Accountable Distillation"—ensuring that vital architectural constraints are never lost # Skill: Architecture Violations and BDI Enforcement

## Purpose

Prevent the silent introduction of architectural debt and layer violations.[16]

## Core Rules

1. **Belief-Desire-Intention (BDI) Parsing:** You must treat all `adr.md` and `constitution.md` files as factual groundings. Your desire is to extract precise constraints. Your intention is to adhere to them flawlessly.
2. **AST-Aware Modification:** When replacing logic, ensure you replace the entire Abstract Syntax Tree (AST) node (the whole function or class), rather than injecting mid-function code that destroys local context.
3. **Layer Isolation:** Presentation logic MUST NOT import Database connection logic directly.

## Anti-patterns

- "Shallow vertical slices": Changing five different architectural concerns across multiple layers in one pass just to make a unit test turn green.[16]

# Skill: Accountable Distillation Discipline

## Purpose

Ensure that when transforming a high-verbosity document into a low-verbosity document, critical architectural capabilities and constraints are tracked.

## Core Rules

1. **Accountable Distillation:** You are permitted to drop conversational fluff and examples. You are FORBIDDEN from dropping security constraints or specific API payload shapes.
2. **The Distillation Loss Statement:** At the end of your generated document, you MUST include a "Distillation Loss Statement".
   - State exactly what context you are dropping from the source material.
   - Justify why the downstream agent does not need this context.[17]

# Skill: Empirical Proof & Show-Don't-Tell

## Purpose

Eliminate agent hallucinations by forcing absolute reliance on standard output streams.

## Core Rules

1. **Never Assume Success:** Writing the code is only 10% of the job. Verifying it works in the current environment is 90%.
2. **Verbatim Pasting:** When filling out the `## Self-review` section of your task file, you MUST paste the verbatim output of the terminal command (e.g., `git diff`, `npm run build`). DO NOT paraphrase by saying "The tests passed."
   06-distillation.md: Accountable Distillation DoctrineThe Swarm framework is conceptually grounded in the doctrine of progressive information distillation. However, as cutting-edge 2026 research indicates, the goal of moving from human intent to agent code is not lossless distillation, but accountable distillation. AI models inherently lose context when compressing research into specifications. Swarm manages this via strict information-loss budgets.The Verbosity Gradient (Diátaxis Mapped) Highest Verbosity (research.md): Contains history, conversational context, and exploratory paths. (Diátaxis "Explanation").Medium Verbosity (spec.md, adr.md, constitution.md): Contains firm rules, exact schemas, and architectural boundaries. All narrative fluff is stripped. (Diátaxis "Reference").Lowest Verbosity (task.md): Highly focused execution vectors. Contains only what is needed to execute a specific, atomic unit of work. (Diátaxis "How-to").The Distillation Loss Statement At each transition down the verbosity gradient, the agent must generate a Distillation Loss Statement. This statement forces the LLM to explicitly acknowledge what data it is choosing not to pass to the next agent in the pipeline.Research → Spec (Loss Budget: High). The Architect agent should drop narratives. The Loss Statement might read: "Dropped: Pricing comparisons of external SDKs. Why: Decision is finalized; the Builder only needs the chosen SDK's endpoint."Audit → Task (Loss Budget: Medium). Suggestive language is dropped.Spec → Task (Loss Budget: Zero). This is a critical transition point. When The Lead Engineer parses a specification into sub-tasks, it is absolutely forbidden from dropping any architectural constraints or data shapes.The Task File Promotion ProtocolTask files are the terminal leaves of the document tree. They are transient, explicitly Git-ignored, and tied to a specific worktree branch that will eventually be deleted.The Protocol:During execution, the agent documents durable discoveries in the ## Decisions, Findings, Assumptions, Blockers, Next steps section of the task.md.Upon task completion, if this section is populated, the manage-task skill forces the agent to append these findings back to the canonical spec.md or spawn a new bug-report.md.Only after this knowledge has been "promoted" up the gradient can the task be marked with status: done.Worked Example: The Verbosity DiffSource Content (High Verbosity research.md):"After reviewing the Stripe documentation and testing the /v1/charges vs /v1/payment_intents endpoints via curl, it's clear we have to migrate to Payment Intents.
